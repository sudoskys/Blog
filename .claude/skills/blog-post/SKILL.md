# Blog Post Skill — sudoskys/Blog

## 项目上下文

- 平台：Hexo，主题 Stellaris / Volantis
- Front matter 模板：`scaffolds/post.md`（title、date、tags，部分帖子有 cover、mermaid、original）
- 发布路径：`source/_posts/<slug>.md`
- 读者：中文技术社区，以 GitHub / 掘金 / 知乎 读者为主
- 风格参考：`ai-era-engineer-growth.md`（中文散文 + 数据驱动，第一人称）；`tanstackdb_with_electricsql.md`（英文技术深潜，Abstract → 架构图 → 逐节展开）

## 一、选题流程

动笔前先过选题清单，任何一条不通过就换选题：

- [ ] **真问题**：有真实读者正被这个问题困扰？（从评论区、Issues、Stack Overflow 验证）
- [ ] **亲身经验**：作者亲自踩过坑并解决？（纯理论推导读者一眼看穿）
- [ ] **足够具体**：标题能精确到工具 / 版本 / 场景？（"深入学习 Docker" ❌ → "Docker 容器日志撑爆磁盘的排查与解决" ✅）
- [ ] **有验证**：方案在真实环境跑通？
- [ ] **时效性**：半年后还有参考价值？（纯新闻类慎选）
- [ ] **差异化**：中文互联网上，至少有一半相关文章没覆盖这个解法？

**痛点金字塔：**

| 读者层级 | 核心痛点 | 适合选题方向 |
|---------|---------|------------|
| 入门（0-1 年） | 跑不起来、配不明白 | "手把手搭建 XXX" |
| 中级（1-3 年） | 踩坑、性能差 | "XXX 踩坑实录 / 性能优化" |
| 高级（3 年+） | 设计取舍、架构演进 | "XXX 的设计取舍与演进" |

## 二、文章结构

**工程级骨架**（比例参考，非硬性字数）：

```
开头   (~10%) — 钩子 + 读者定位 + 文章承诺，三句内完成
背景   (~15%) — 为什么已有方案不够，走过哪些弯路
主体   (~50%) — 架构决策 + 代码 + 踩坑 + 权衡取舍，每节只解决一个问题
结果   (~15%) — 指标、前后对比、真实数字
结尾   (~10%) — 核心要点 + 边界 + 下一步
```

**开头要完成三件事**（三句内）：
1. 这篇写的是什么
2. 为什么值得看
3. 读者看完能带走什么

四种有效开头（选一）：
- **具体场景型**：直接描述触发文章的那个时刻（"有一天我意识到..."）
- **认知修正型**：先说原来怎么想，再说实际踩坑后发现什么
- **反直觉数据型**：一个让目标读者停下来的数字或结论
- **先给判断型**：开头给结论，再展开理由和边界

❌ 禁止的开头：
- "在当今快速发展的技术世界中..."
- "随着 XX 技术的不断演进..."
- "近年来，XXX 受到越来越多的关注..."
- 先铺大背景再慢慢靠近主题

**结尾四种收法**（选一）：
1. 一句判断收尾
2. 给读者下一步行动
3. 补边界（"这个方案不适用于..."）
4. 回扣开头问题

❌ 禁止的结尾：突然上价值、把结论写成口号、空泛复述全文

**标题公式：** 动词 + 具体结果 + 技术/场景上下文

```
[解决/排查/从X迁移到Y] + [具体问题] + [版本/工具/场景]
```

示例：
- "排查 Kafka 消费者积压：从告警到根因的完整过程"
- "把 ElectricSQL + TanStack DB 接入 React：乐观更新的正确姿势"

## 三、句子层面写作

### 3.1 密度与节奏

**一段只做一件事**：下定义、解释机制、展示证据、下判断——不要乱炖。

**长句装信息，短句落锤**：长句交代背景/边界/对象，短句下判断。读者需要呼吸。

```
❌ 单一节奏（全是短句打电报）：
这个问题很常见。很多人都踩过。解法其实简单。只需要改一行配置。

✅ 混合节奏（长信息 + 短落锤）：
这个问题的根源在于 Docker 默认不限制 json-file 日志驱动的大小——
容器跑得越久，日志文件就越大，直到把磁盘撑爆。
改一行配置就能解决。
```

**句子长度多样化**：短句制造冲击，长句构建逻辑，二者交替使用。连续四五个句号 = 打电报，合并或用逗号连。

**核心观点全文只出现 2 次**：定义时 + 总结时。中间段落直接使用，不再反复解释。删掉后不影响理解的段，就是废话，删。

### 3.2 具体胜于抽象（Show vs. Tell 的技术文章版）

写作的最大陷阱：用抽象词躲避精确性。

```
❌ 抽象："这个方案有很好的性能"
✅ 具体："压测显示 P99 延迟从 340ms 降到 47ms"

❌ 抽象："遇到了一个奇怪的 bug"
✅ 具体："ECONNREFUSED 127.0.0.1:5432，但容器确实在跑"

❌ 抽象："这个框架很流行"
✅ 具体："React 19.1，GitHub 230k stars"
```

具体化的方法：
- 把形容词换成数字（"快" → "快 3 倍"）
- 把"某个" 换成具名对象（"某个流行框架" → "Next.js 15"）
- 把结论换成可观测的现象（"性能好" → "在 M2 MacBook 上 cold start 800ms"）
- 给每个代码示例加上它要证明的一句话

### 3.3 去 AI 腔

AI 生成的中文技术文有高度可识别的模式。写完后扫一遍：

**最高频问题（优先检查）：**
- 段末有「这说明」「可以看出」「到这里」「通过以上分析」开头的收尾总结句 → **直接删**，上一句已经说完了
- 有「不是...而是...」对比句式 → 只说结论：「不是 X 而是 Y」→「是 Y」
- 升华句（把具体工程观察拔高到人生哲理）→ **删**，换成具体建议

**禁用词：**

| 不用 | 改成 |
|------|------|
| 随着...的不断发展 | （删，直接开始） |
| 值得注意的是 | （直接说结论） |
| 综上所述 / 总而言之 | （直接收尾） |
| 非常 / 极其 | 很 |
| 例如 | 比如 |
| 赋能 / 抓手 / 闭环 / 沉淀 / 对齐 | 改成具体动词 |
| 很多同学 | 很多人 |
| 至关重要 / 意义重大 | 重要 |
| 进行（+ 动词）| 直接用动词 |

**禁用句式：**
- 自问自答 → 直接陈述
- 三连排比连续出现两次 → 换二连或一连
- 开篇「你好！欢迎...」→ 直接开始
- 章节间「上面讲了...下面我们看...」过渡句 → 删

**标点：**
- 用直角引号 `「」`，不用弯引号 `""`
- 禁用破折号（—），改逗号或句号
- 不用感叹号（除非真有情绪需要表达）
- 中文和英文/数字之间加空格：「Python 爬虫」「版本 2.0」

### 3.4 术语处理

判断标准：中文技术圈有没有公认译法？有就用中文，没有就保留英文。

已有公认译法的用中文（上下文、状态、缓存、运行时等）。仍在抢的保留英文（prompt、embedding、tokenizer、agent 作为框架组件名时）。已成专有名词的保留英文（React、Docker、Kafka 等）。

第一次出现的新概念，先给一句人话解释，再展开。

### 3.5 写作过程

先写烂草稿捕捉逻辑，再重写塑形。写完再改，不要边写边打磨（会误以为完成的段不能动）。最终做大量删减：删掉后不影响理解的段就是废话。

## 四、代码规范

代码块前必须有「用途句」：这段要证明什么，读者重点看哪里。

```typescript
// ✅ 有用途句的示例：
// 关键在第 8 行：我们在提交前先做了乐观更新，
// 失败时 TanStack DB 会自动回滚，组件不需要手动处理错误状态。
const result = await collection.insert({...})
```

代码块要求：
- 必须标注语言（` ```typescript`、` ```bash` 等）
- 单块不超过 30 行；更长的拆分并加注释
- 包含版本号（依赖版本、OS 版本、运行时版本）
- 展示预期输出或错误信息，不只展示输入
- 代码能运行的给可运行版本；不能运行的明确说明是「示意 / 伪代码 / 节选」
- 不编造不存在的 API / 类名 / 方法名

## 五、可信度信号

以下信号让文章有别于 AI 生成的平庸内容。至少包含 2 条：

- **展示死胡同**：「我先试了 X，因为 Y 失败了」
- **暴露真实错误**：粘贴真实的 stack trace 或报错，不是「会报一个错误」
- **承认不确定**：「我还不清楚为什么这里要这样设计」
- **真实数字**：「在我的 M2 MacBook 上跑了 47 分钟」「P99 从 340ms 降到 47ms」
- **版本具体**：不写「某个流行框架」，写「React 19.1」
- **时间戳**：「凌晨两点排查的时候发现...」
- **工具痕迹**：具体工具名、命令、trace 观察点

## 七、发布前审查清单

### L1 硬性规则（不通过不发布）

- [ ] 开头三句内完成：这是什么 + 为什么值得看 + 读者能带走什么
- [ ] 没有「随着...的发展」「在当今...时代」等开篇套话
- [ ] 没有段末「这说明...」「通过以上分析...」总结句
- [ ] 没有「不是...而是...」对比句式
- [ ] 没有赋能 / 抓手 / 闭环 / 沉淀等黑话
- [ ] 所有代码块有语言标注
- [ ] 中文和英文 / 数字之间有空格

### L2 可读性

- [ ] 段落不超过 4 句，句子长度有变化
- [ ] 核心观点没有反复重申
- [ ] 没有升华句（具体观察 → 人生哲理）
- [ ] 删掉之后不影响理解的段已删除

### L3 内容深度

- [ ] 至少包含 2 条可信度信号（死胡同 / 真实错误 / 真实数字等）
- [ ] 代码块前有用途句，后有解释句
- [ ] 结论明确并附有依据和适用边界

### L4 活人感

- [ ] 读起来像工程师给同事讲系统，不像报告也不像口播稿
- [ ] 没有连续三个以上三连排比
- [ ] 任意摘出一段，语气和用词与全文一致（无突然变正式或变随意的段落）

## 八、工作流程

**写新博文：**
1. 过选题清单，确认选题有效
2. 确认目标读者层级（痛点金字塔）
3. 输出结构大纲供确认
4. 按工程级骨架撰写，遵循第三节句子层面规范
5. 过发布前审查清单（L1 必须全通过）
6. 输出完整 front matter + 正文，可直接保存为 `source/_posts/<slug>.md`

**审查现有博文：**
1. 阅读全文
2. 按 L1→L2→L3→L4 逐层检查
3. 给出具体改进建议，标注行号或引用原句

**润色 / 去 AI 味：**
1. 先识别文章类型（技术深潜 / 散文观点 / 教程）
2. 按 3.3 节扫描禁用词和禁用句式
3. 做减法，不做加法：先删总结腔 / 对比句式 / 黑话，再调节奏
4. 不改代码字面量、URL、API 路径、技术专有名词
5. 如果原句已经自然准确，不改
