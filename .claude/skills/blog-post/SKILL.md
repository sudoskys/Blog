# Blog Post Skill — sudoskys/Blog

## 项目上下文

- 平台：Hexo，主题 Stellaris / Volantis
- Front matter 模板：`scaffolds/post.md`（title、date、tags，部分帖子有 cover、mermaid、original）
- 发布路径：`source/_posts/<slug>.md`
- 读者：中文技术社区，以 GitHub / 掘金 / 知乎 读者为主
- 风格参考：`ai-era-engineer-growth.md`（中文散文 + 数据驱动，第一人称）；`tanstackdb_with_electricsql.md`（英文技术深潜，Abstract → 架构图 → 逐节展开）

## 一、文体分类

**写之前先确认文体**，后面的结构规范按文体选取：

| 文体 | 核心目标 | 典型触发 | 跳到 |
|------|---------|---------|------|
| **技术文章** | 帮读者解决具体问题 / 理解机制 | "怎么做 X"、"踩坑记录"、"架构演进" | §二 |
| **观点/杂谈** | 让读者接受或认真考虑一个主张 | "我的看法"、"有感而发"、"杂谈"、"聊聊 X" | §三 |
| **叙事/随笔** | 让读者感受一个经历或情绪 | "记录一次经历"、"随笔"、"回顾"、文学性描述 | §四 |

混合文体（如「通过一次经历谈工程设计」）：叙事开头，技术为主体，观点收尾——
开头遵循 §四的具体时刻原则，主体遵循 §二的骨架，结尾遵循 §三的判断收法。

## 二、技术文章

### 选题清单

动笔前先过，任何一条不通过就换选题：

- [ ] **真问题**：有真实读者正被这个问题困扰？（从评论区、Issues、Stack Overflow 验证）
- [ ] **亲身经验**：作者亲自踩过坑并解决？（纯理论推导读者一眼看穿）
- [ ] **足够具体**：标题能精确到工具 / 版本 / 场景？（"深入学习 Docker" ❌ → "Docker 容器日志撑爆磁盘的排查与解决" ✅）
- [ ] **有验证**：方案在真实环境跑通？
- [ ] **时效性**：半年后还有参考价值？（纯新闻类慎选）
- [ ] **差异化**：中文互联网上，至少有一半相关文章没覆盖这个解法？

**痛点金字塔：**

| 读者层级 | 核心痛点 | 适合选题方向 |
|---------|---------|------------|
| 入门（0-1 年） | 跑不起来、配不明白 | "手把手搭建 XXX" |
| 中级（1-3 年） | 踩坑、性能差 | "XXX 踩坑实录 / 性能优化" |
| 高级（3 年+） | 设计取舍、架构演进 | "XXX 的设计取舍与演进" |

### 工程级骨架

（比例参考，非硬性字数）：

```
开头   (~10%) — 钩子 + 读者定位 + 文章承诺，三句内完成
背景   (~15%) — 为什么已有方案不够，走过哪些弯路
主体   (~50%) — 架构决策 + 代码 + 踩坑 + 权衡取舍，每节只解决一个问题
结果   (~15%) — 指标、前后对比、真实数字
结尾   (~10%) — 核心要点 + 边界 + 下一步
```

**开头要完成三件事**（三句内）：
1. 这篇写的是什么
2. 为什么值得看
3. 读者看完能带走什么

❌ 禁止的开头：
- "在当今快速发展的技术世界中..."
- "随着 XX 技术的不断演进..."
- 先铺大背景再慢慢靠近主题

**标题公式：** 动词 + 具体结果 + 技术/场景上下文

```
[解决/排查/从X迁移到Y] + [具体问题] + [版本/工具/场景]
```

示例：
- "排查 Kafka 消费者积压：从告警到根因的完整过程"
- "把 ElectricSQL + TanStack DB 接入 React：乐观更新的正确姿势"

## 三、观点文/杂谈

### 核心原则

观点文的目标不是「分享」，是「说服」——让读者看完之后，接受或认真考虑你的主张。
没有主张的「聊聊」是漫谈，漫谈不需要结构，但也很难让人记住。

### CER 框架（每个主论点）

**Claim → Evidence → Reasoning**

1. **Claim（主张）**：一句话，具体可反驳的论断。不是「设计很重要」，是「大多数系统在初期就决定了它能否扩展」。
2. **Evidence（证据）**：支撑这个主张的具体事实。个人经历、可观察的数据、引用的案例——三选一，但必须具体。
3. **Reasoning（推理）**：从证据到结论的逻辑步骤。读者看完应该能自己重复这个推理，不靠信任你。

```
❌ 漫谈式：
"我觉得大家对 AI 工具过于依赖了。很多人写代码都要问 AI，其实这样不太好。"

✅ CER 式：
"过早引入 AI 辅助会减慢初级工程师的技能积累速度（主张）。
原因是：错误信号消失了——AI 会把语法错误悄悄修掉，
让工程师不知道自己在哪些基础问题上有盲区（推理）。
我见过一个连 null pointer 都看不出来但 AI 用得很溜的实习生（证据）。"
```

### 反驳段

**每篇观点文至少包含一个反驳段。**

反驳段的作用：让文章有张力，同时证明你认真想过另一面。
写法：先诚实描述对立观点，然后解释为什么你的主张仍然成立（或成立的条件）。

```
❌ 假反驳（设稻草人然后推倒）：
"当然，有人可能会说 AI 不好，但那是因为他们没用对。"

✅ 真反驳：
"反对者说得有道理：AI 工具确实节省了大量查文档的时间，
对熟练工程师来说是净收益。
我的限定条件是「初级工程师在技能未稳固期」——
这个阶段查文档本身就是学习路径，跳过会留下盲区。"
```

### 观点文结构

```
开头   — 一个让目标读者停下来的具体观察/场景（不是"我想聊聊X"）
主张   — 明确的一句话论断（最好在第二段就亮出来）
论证   — 2-3 个 CER 段落，每段一个论据，不重叠
反驳   — 诚实的一段，不是点缀
结尾   — 收窄：这个主张适用的边界 + 一个具体的行动建议或判断
```

### 螺旋深化（超越单次 CER）

整篇文章不是「论点→证据→结论」一次完成，而是同一论点在不同层面反复出现，每次更深一层：
第一次是现象观察，第二次是机制解释，第三次是边界条件。

**2+1 框架**（绕过读者心理防线）：
1. 展示常见错误做法（解释为何失败）
2. 展示诱人的捷径（解释为何适得其反）
3. +1 正确路径——此时你的方案成为「唯一合理选择」

**姿态**：「同行者」而非「老师」。呈现的是「我和你站在同一困境里，刚蹚出一小段路」。

（更多说服技术见 `references/writing-craft.md` §观点文螺旋说服）

❌ 禁止的结尾：突然「上价值」（"让我们一起努力..."）、把主张重复一遍当总结、模糊的感悟

## 四、叙事/随笔

### 核心原则

叙事的目标是让读者感受，不是让读者知道。知道靠陈述，感受靠细节。
「我那天很疲惫」是陈述。「凌晨两点，键盘上有我手肘压出的痕迹」是细节。

### 具体时刻

叙事文章从「具体时刻」开始，不从「一段时间」或「一个阶段」开始。

```
❌ 模糊时段：
"在那段创业期间，我学到了很多东西..."

✅ 具体时刻：
"2023 年 11 月的某个下午，我坐在咖啡馆里，盯着屏幕上那封退款申请..."
```

具体时刻的三个要素：
- **时间点**（几月、几点、什么节气）
- **空间**（在哪儿、周围什么环境）
- **动作**（你在做什么，不是你在感受什么）

### 细节三层

一个好细节应该同时在三个层面上工作：

| 层 | 问题 | 示例 |
|----|------|------|
| **形境**（可观察的感官细节） | 读者能看到/听到/闻到什么？ | "屏幕亮度调到最低，风扇声是房间里唯一的声音" |
| **情境**（人物的情绪/判断） | 角色此刻感受到什么？ | "我知道再等十分钟什么结果都没有了" |
| **理境**（读者能带走的意义） | 这个细节说明了什么更大的事情？ | （不必明说，让形境和情境共同暗示） |

**理境通常不用写出来**。写出来反而变成说教。让读者自己在形境和情境中找到它。

```
❌ 三层都说出来（过度解释）：
"我看着空空的账户（形境），心里很难受（情境），
这让我意识到创业的残酷性（理境）。"

✅ 形境 + 情境，理境隐含：
"账户余额显示：¥247.00。我没有告诉任何人。"
```

### 叙事结构选项

- **线性顺叙**：按时间顺序，适合「记录一次经历」
- **闪回式**：从现在某个时刻出发，回溯关键节点，适合「回顾」
- **并行对比**：两条线索交替推进，适合「对比两种状态/选择」
- **快照式**：几个不相连的具体场景，用空行或标题隔开，适合情绪散文

随笔不需要「完整」——可以没有结论，可以以一个场景收尾而不做总结。
让最后一个场景或句子承担意义，不用「后来我意识到...」来解释它。

### 节奏控制

博客叙事（1500-5000 词）在三种模式间切换：

| 模式 | 功能 | 时间效果 |
|------|------|----------|
| **场景** | 沉浸感、即时性 | 放慢时间（秒对秒展开） |
| **摘要** | 压缩、移动 | 加速时间（一句话覆盖数月） |
| **反思** | 开出意义 | 暂停时间（穿插在场景中） |

**速度变化 = 重要性信号**：两句话总结整个上午，然后在打开关键邮件的瞬间切换到实时。
读者不需要你说「注意，这里重要」——速度本身就在说。

场景选择标准：1500-5000 词选 3-5 个场景，每个必须推进变化、提升 stakes、或揭示新信息。
用摘要桥接场景，但摘要中也要有具体细节锚点。

（更多节奏技巧见 `references/writing-craft.md` §叙事节奏）

## 五、通用写作规范

（适用于所有文体。更深入的技巧见 `references/writing-craft.md`）

### 5.1 密度与节奏

**一段只做一件事**：下定义、解释机制、展示证据、下判断——不要乱炖。

**长句装信息，短句落锤**：长句交代背景/边界/对象，短句下判断。

```
❌ 单一节奏（全是短句打电报）：
这个问题很常见。很多人都踩过。解法其实简单。只需要改一行配置。

✅ 混合节奏（长信息 + 短落锤）：
这个问题的根源在于 Docker 默认不限制 json-file 日志驱动的大小——
容器跑得越久，日志文件就越大，直到把磁盘撑爆。
改一行配置就能解决。
```

**核心观点全文只出现 2 次**：定义时 + 总结时。删掉后不影响理解的段，就是废话，删。

### 5.2 具体胜于抽象

```
❌ "这个方案有很好的性能"
✅ "压测显示 P99 延迟从 340ms 降到 47ms"

❌ "遇到了一个奇怪的 bug"
✅ "ECONNREFUSED 127.0.0.1:5432，但容器确实在跑"
```

具体化方法：把形容词换成数字、把「某个」换成具名对象、把结论换成可观测现象。

### 5.3 去 AI 腔

**最高频问题（优先检查）：**
- 段末有「这说明」「可以看出」「到这里」「通过以上分析」开头的收尾总结句 → **直接删**
- 有「不是...而是...」对比句式 → 只说结论
- 升华句（把具体工程观察拔高到人生哲理）→ **删**，换成具体建议

**禁用词：**

| 不用 | 改成 |
|------|------|
| 随着...的不断发展 | （删，直接开始） |
| 值得注意的是 | （直接说结论） |
| 综上所述 / 总而言之 | （直接收尾） |
| 非常 / 极其 | 很 |
| 赋能 / 抓手 / 闭环 / 沉淀 / 对齐 | 改成具体动词 |
| 很多同学 | 很多人 |
| 至关重要 / 意义重大 | 重要 |
| 进行（+ 动词）| 直接用动词 |

**禁用句式：**
- 自问自答 → 直接陈述
- 三连排比连续出现两次 → 换二连或一连
- 章节间「上面讲了...下面我们看...」过渡句 → 删

**标点：**
- 用直角引号 `「」`，不用弯引号 `""`
- 禁用破折号（—），改逗号或句号
- 中文和英文/数字之间加空格：「Python 爬虫」「版本 2.0」

### 5.4 视觉节奏

- **每 300-400 词一次视觉断裂**（图片、列表、callout、表格、代码块均可）
- **密集元素后跟呼吸器**：大表格或代码块之后，放一句总结或留白
- **段落 1-3 句**（手机上 4 行以上是文字墙）
- **Squint test**：把页面缩到看不清字，应该看到深浅交替的块，不是一堵墙

### 5.5 段落衔接（不用过渡句）

禁止模板化过渡（"首先/其次"、"接下来让我们"），用以下技术替代：
- **回声词**：上段关键词在下段开头自然出现
- **尾-头钩锁**：前句末尾信息 = 下句开头信息
- **时间/空间锚**：通过场景、时间流动暗示过渡

检验：去掉标题后，纯文本仍然流畅 = 衔接成功。
（完整六种技术见 `references/writing-craft.md` §无形过渡）

### 5.6 术语处理

中文技术圈有公认译法的用中文（上下文、状态、缓存、运行时等）。
仍在抢的保留英文（prompt、embedding、tokenizer、agent 作为框架组件名时）。
已成专有名词的保留英文（React、Docker、Kafka 等）。
第一次出现的新概念，先给一句人话解释，再展开。

### 5.7 开头硬约束

- 引言 80-120 词（过短缺上下文，过长延迟价值承诺）
- 第一句 < 20 词
- 以「我」或具体场景开头 → 比「How to」「The」开头获得 3 倍以上参与

## 六、代码规范

（仅技术文章适用）

代码块前必须有「用途句」：这段要证明什么，读者重点看哪里。

```typescript
// ✅ 有用途句的示例：
// 关键在第 8 行：我们在提交前先做了乐观更新，
// 失败时 TanStack DB 会自动回滚，组件不需要手动处理错误状态。
const result = await collection.insert({...})
```

代码块要求：
- 必须标注语言（` ```typescript`、` ```bash` 等）
- 单块不超过 30 行；更长的拆分并加注释
- 包含版本号（依赖版本、OS 版本、运行时版本）
- 展示预期输出或错误信息，不只展示输入
- 不编造不存在的 API / 类名 / 方法名

## 七、可信度信号

以下信号让文章有别于 AI 生成的平庸内容。**至少包含 2 条**：

- **展示死胡同**：「我先试了 X，因为 Y 失败了」
- **暴露真实错误**：粘贴真实的 stack trace 或报错
- **承认不确定**：「我还不清楚为什么这里要这样设计」
- **真实数字**：「在我的 M2 MacBook 上跑了 47 分钟」
- **版本具体**：不写「某个流行框架」，写「React 19.1」
- **时间戳**：「凌晨两点排查的时候发现...」
- **工具痕迹**：具体工具名、命令、trace 观察点

## 九、发布前审查清单

### L1 硬性规则（不通过不发布）

- [ ] 开头三句内完成：这是什么 + 为什么值得看 + 读者能带走什么（技术文）；或以具体时刻开始（叙事）
- [ ] 没有「随着...的发展」「在当今...时代」等开篇套话
- [ ] 没有段末「这说明...」「通过以上分析...」总结句
- [ ] 没有「不是...而是...」对比句式
- [ ] 没有赋能 / 抓手 / 闭环 / 沉淀等黑话
- [ ] 所有代码块有语言标注（技术文）
- [ ] 中文和英文 / 数字之间有空格

### L2 可读性

- [ ] 段落不超过 4 句，句子长度有变化
- [ ] 核心观点没有反复重申
- [ ] 没有升华句（具体观察 → 人生哲理）
- [ ] 删掉之后不影响理解的段已删除

### L3 内容深度

- [ ] 至少包含 2 条可信度信号
- [ ] 技术文：代码块前有用途句，后有解释句
- [ ] 观点文：每个主要论点有 Evidence，有至少一段反驳
- [ ] 叙事文：有具体时刻开头，细节至少有形境 + 情境两层

### L4 活人感

- [ ] 读起来像工程师给同事讲系统（技术文）/ 像朋友讲故事（叙事）/ 像在认真争论（观点）
- [ ] 没有连续三个以上三连排比
- [ ] 任意摘出一段，语气和用词与全文一致

## 十、工作流程

**写新博文：**
1. 确认文体（技术/观点/叙事），参考 §一
2. 过对应文体的结构要求（§二 / §三 / §四）
3. 输出结构大纲供确认
4. 按骨架撰写，遵循 §五通用规范
5. 过 L1 审查清单（必须全通过）
6. 输出完整 front matter + 正文，可直接保存为 `source/_posts/<slug>.md`

**审查现有博文：**
1. 阅读全文，先识别文体
2. 按 L1→L2→L3→L4 逐层检查（L3 按文体选对应条目）
3. 给出具体改进建议，标注行号或引用原句

**润色 / 去 AI 味：**
1. 先识别文体
2. 按 §5.3 扫描禁用词和禁用句式
3. 做减法：先删总结腔 / 对比句式 / 黑话，再调节奏
4. 不改代码字面量、URL、API 路径、技术专有名词
5. 原句已经自然准确的，不改
